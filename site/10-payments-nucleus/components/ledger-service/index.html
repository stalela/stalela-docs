    Ledger Service¶

    The Ledger Service is Stalela‚Äôs book of record. It maintains double‚Äëentry journals, computes balances, exposes statements, and guarantees that every money movement is represented by a balanced set of postings. If it isn‚Äôt posted here, it didn‚Äôt happen.

    üéØ Purpose¶

    • Enforce double‚Äëentry accounting for all transfers across rails.
    • Provide authoritative balances and auditable journals.
    • Support authorization ‚Üí capture ‚Üí reversal/return lifecycles.
    • Handle multi‚Äëcurrency with explicit FX conversion and P&L.
    • Export partner statements (BAI2‚Äëlike) and support reconciliation.

    üõ† Responsibilities¶

    • Consume domain events and apply posting rules:
    • transfers.accepted (hold / memo postings where applicable)
    • transfers.settled (final postings)
    • transfers.returned (reversal postings)
    • Maintain derived balances (by account, currency).
    • Emit ledger.balance.updated and ledger.posting.created.
    • Provide read APIs: balances, journals, statements.
    • Guard rails: append‚Äëonly journals; reversals via new entries only.

    üîå Interfaces¶

    Events (consume; envelope v=1)¶

    • transfers.accepted
    • transfers.settled
    • transfers.returned

    Dedupe per Events spec: see ../specs/events.md (consumer idempotency by eventId, lifecycle uniqueness for (transferId,type)).

    Events (emit; envelope v=1)¶

    • ledger.posting.created
    • ledger.balance.updated

    HTTP (read‚Äëonly)¶

    • GET /balances?accountId=...&currency=...
    • GET /journal?transferId=... or ?accountId=...&from=...&to=...
    • GET /statements?accountId=...&from=...&to=...&format=bai2|json

    Admin (via Platform/Base)¶

    • GET /live, GET /ready, GET /metrics, GET /version provided by Platform/Base and adopted by Ledger.

    Write access is event‚Äëdriven only. Manual adjustments are rare and must go through the Operator Console using explicit adjustment events with approvals/audit.

    üóÑ Data Model¶

    tables - accounts
    - accountId (pk), type (USER|MERCHANT|LIQUIDITY|FEES|FX|SETTLEMENT|RESERVE|NOSTRO|VOSTRO), tenantId?, currency? (nullable if multi‚Äëcurrency), status, createdAt - journals (immutable)
    - journalId (pk), transferId, eventType, occurredAt, memo, exchangeControlRef? - postings (append‚Äëonly)
    - postingId (pk), journalId (fk), debitAccountId, creditAccountId, amountMinor, currency, memo - balances (materialized)
    - accountId, currency, balanceMinor, asOf - outbox_ledger ‚Äì for emitted events

    blobs (encrypted)
    - optional statement snapshots; export artifacts

    üßÆ Posting Rules (canonical)¶

    See: ../specs/posting-rules.md

    All amounts are in minor units (e.g., cents). Examples assume a customer pays a merchant 100.00 with a 1.00 fee.

    1) Accepted (authorization / hold) ‚Äî rails that support AUTH/CAPTURE¶

    Create memo/hold journal (off‚Äëbalance or flagged): - Debit: USER_AUTH_HOLD (memo) ‚Äî 10000
    - Credit: LIQUIDITY_AUTH_PENDING (memo) ‚Äî 10000

    If the rail has no auth concept (e.g., USDC push), no accepted postings are created.

    2) Settled (final funds move)¶

    Core settlement postings: - Debit: USER ‚Äî 10000
    - Credit: MERCHANT ‚Äî 9900
    - Credit: FEES ‚Äî 100 (Stalela take rate)

    If FX occurs (payer USD ‚Üí merchant ZAR), split legs: - Debit: USER ‚Äî 10000 USD
    - Credit: LIQUIDITY ‚Äî 10000 USD
    - Debit: LIQUIDITY ‚Äî 180000 ZAR
    - Credit: MERCHANT ‚Äî 180000 ZAR
    - Credit/Debit: FX_PNL ‚Äî difference from quoted vs realized

    Cross‚Äëborder flows may include legs to/from NOSTRO/VOSTRO accounts to reflect settlement with external banks.

    VAT on Fees (ZA)¶

    • When fees are vatable, split into FEES_NET and FEES_VAT credits as per 20-specs/tax-vat.md.

    3) Returned / Chargeback (reverse prior settled)¶

    Post the exact contra of the settlement set, preserving original currency: - Debit: MERCHANT ‚Äî 9900
    - Debit: FEES ‚Äî 100 (or use FEES_REVERSAL to separate)
    - Credit: USER ‚Äî 10000

    For partial returns, amounts reflect the returned portion; keep link to original journal via relatedJournalId (in journals.memo).

    4) Fees & Surcharges¶

    • Percentage fees applied at settlement time as separate credit to FEES account.
    • Per‚Äërail surcharges (e.g., OPPWA) post to FEES_PARTNER account and are netted in reconciliation.

    5) Rounding¶

    • Use banker‚Äôs rounding when converting decimals ‚Üí minor units.
    • Any rounding residual posts to FX_PNL_ROUNDING (must be near‚Äëzero).

    üìê Diagrams¶

    Event ‚Üí Posting pipeline¶

    flowchart LR
  EV[transfers.* event] --> PR[Posting Rules]
  PR --> JR[Create Journal]
  JR --> PT[Append Postings (balanced)]
  PT --> BL[Update Balances]
  PT --> OE[Emit ledger.posting.created]
  BL --> BE[Emit ledger.balance.updated]

    Settlement example (no FX)¶

    sequenceDiagram
  participant GW as Rail Gateway
  participant L as Ledger
  participant S as Balance Store

  GW-->>L: transfers.settled {amount=10000, fee=100}
  L->>L: create journal J1
  L->>L: postings: D USER 10000 / C MERCHANT 9900 / C FEES 100
  L->>S: update balances (USER-, MERCHANT+, FEES+)
  L-->>GW: ledger.posting.created (J1)

    üß± Invariants¶

    • Balanced: sum(debits) == sum(credits) per journal.
    • Append‚Äëonly: no deletes; corrections via new journals.
    • Idempotent: journal key = hash(transferId,eventType,sequence); duplicates ignored.
    • Temporal: occurredAt from event; do not rewrite history.

    Consumer idempotency: dedupe inbound events by eventId, and enforce lifecycle uniqueness (transferId,eventType) to avoid double-posting.

    üö® Failure Modes & Retries¶

    • Duplicate event ‚Üí ignored via idempotency key.
    • Currency mismatch (account vs posting) ‚Üí reject and raise operator alert.
    • Negative balance constraints (if configured) ‚Üí block posting and raise exception.
    • Outbox publish failure ‚Üí retry with backoff; no partial commits (journal + outbox in same txn).

    üìä Observability¶

    • Metrics: journals/sec, postings/sec, balance update latency, idempotency drop rate.
    • Gauges: outstanding reversal backlog, FX_PNL daily total.
    • Logs: structured with transferId, journalId, accountId, currency.
    • Traces: span across event reception ‚Üí journal write ‚Üí outbox publish.

    üîê Security¶

    • Least privilege DB role; append‚Äëonly constraints at the schema level if possible.
    • PII: account metadata stored outside postings; journals contain IDs only.
    • Statement exports signed + timestamped; encrypt artifacts at rest.
    • Admin endpoints behind service mesh authN/Z; metrics unauthenticated readiness only.

    ‚öôÔ∏è Config¶

    • LEDGER_BASE_CURRENCY (reporting), supported currencies.
    • NEGATIVE_BALANCE_POLICY (ALLOW|BLOCK|WARN).
    • FX_QUOTE_SOURCE (provider, TTL).
    • STATEMENT_FORMATS (bai2,json), STATEMENT_TIMEZONE.
    • IDEMPOTENCY_SALT for journal keys.

    üìò Runbooks¶

    • Balances off vs partner ‚Üí run T+0/T+1 reconciliation; inspect unmatched queue; verify posting rules version.
    • FX_PNL spikes ‚Üí check quote source latency; ensure correct pair & decimals.
    • Reversal backlog ‚Üí confirm returns events being emitted; investigate gateway DLQs.
    • Throughput degradation ‚Üí check DB contention on balances; consider batched/materialized updates.

    üìè SLOs¶

    • Posting latency (P99) ‚â§ 1s after transfers.settled.
    • Idempotency duplicate acceptance rate ‚â§ 0.1%.
    • Balance read latency (P99) ‚â§ 200 ms.
    • Statement generation ‚â§ 60 s for 100k postings.

    üß™ Test Matrix (essentials)¶

    • Round‚Äëtrip: accepted ‚Üí settled ‚Üí returned (full & partial).
    • Multi‚Äëcurrency settlement with FX & rounding residuals.
    • Idempotency under concurrent duplicates.
    • Negative balance policy enforcement.
    • Statement export byte‚Äëfor‚Äëbyte determinism for a fixed fixture.
